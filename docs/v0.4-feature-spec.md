# NxQuantum v0.4 Feature Specification

## Release Theme

Dynamic execution, hardware bridge contracts, and scale/performance/product positioning maturity.

## Why v0.4 Matters

1. Move beyond static-circuit boundaries into practical dynamic-circuit execution.
2. Establish provider-facing contracts required for real hardware workflows.
3. Improve scale/performance posture and publish competitive evidence for ML teams evaluating NxQuantum vs Python-first alternatives.

## Priority Selection (Pareto)

1. **P0: Dynamic execution + provider bridge contracts**.
2. **P1: Scale and performance maturity**.
3. **P2: Product positioning and adoption assets**.

## Feature Set

### F1. Dynamic Execution + Hardware Bridges [P0]

Goal:

1. Execute an approved dynamic IR subset and define deterministic provider lifecycle boundaries.

Scope:

1. Runtime execution for supported dynamic IR nodes (measurement + conditional feed-forward subset).
2. Provider-facing job lifecycle contract (`submit`, `poll`, `cancel`, `fetch_result`).
3. Deterministic provider error mapping into typed NxQuantum errors.
4. Calibration payload contract hooks for mitigation pipelines.

Acceptance:

1. Supported dynamic IR graphs execute deterministically for fixed seed/input/profile.
2. Provider lifecycle transitions are typed and reproducible.
3. Invalid provider/calibration payloads return typed diagnostics.

### F2. Scale and Performance Maturity [P1]

Goal:

1. Improve runtime scalability and make performance evidence reproducible.

Scope:

1. Large-scale simulator fallback strategy (tensor-network/MPS-oriented path).
2. Batched execution optimization targets for `batch >= 32`.
3. Cross-profile benchmark matrix publication (latency/throughput/memory).
4. CI regression thresholds for performance-sensitive contracts.

Acceptance:

1. Fallback paths are deterministic and configurable via typed options.
2. Batched outputs preserve scalar-reference equivalence while improving throughput on target lanes.
3. Benchmark reports are reproducible and versioned.

### F3. Product Positioning vs Python-First Alternatives [P2]

Goal:

1. Ship evidence-driven positioning assets for adoption decisions.

Scope:

1. Side-by-side workflow guides (NxQuantum vs Python-first tooling) for representative tasks.
2. Migration playbooks from common Python workflows to NxQuantum APIs.
3. Livebook-first tutorials with runnable sample datasets.
4. Explicit decision matrix including both NxQuantum-fit and Python-first-fit scenarios.
5. Case-study benchmark narratives tied to reproducible scripts.

Acceptance:

1. Positioning docs are explicit, honest about limitations, and backed by reproducible data.
2. Migration/tutorial assets are runnable and deterministic.
3. Claims in positioning assets map to benchmark or feature-test evidence.

## Feature Coverage Plan (v0.4)

1. `features/dynamic_execution_hardware_bridges.feature`
2. `features/scale_and_performance.feature`
3. Positioning artifacts under docs/tutorials (non-executable by design).

Planned step mappings:

1. `test/features/steps/dynamic_execution_hardware_bridges_steps.ex`
2. `test/features/steps/scale_and_performance_steps.ex`
3. No BDD step module for positioning-only assets (tracked via docs deliverables).

## Delivery Plan

### Milestone F (Phase 6 / P0)

1. Dynamic execution subset implementation + provider lifecycle contracts.
2. Calibration payload and typed hardware error mapping.

### Milestone G (Phase 7 / P1)

1. Scale fallback strategy + batch execution performance targets.
2. Reproducible benchmark matrix and CI regression gates.

### Milestone H (Phase 8 / P2)

1. Positioning docs, migration guides, Livebook tutorials.
2. Decision matrix and evidence-backed case-study narratives.

## Risks and Mitigations

1. Risk: dynamic execution scope creep.
   Mitigation: enforce subset-only execution boundary per milestone.
2. Risk: performance claims without reproducible evidence.
   Mitigation: require versioned benchmark scripts/reports for every claim.
3. Risk: positioning drift into marketing-only narrative.
   Mitigation: tie every claim to test/benchmark or explicit limitation statement.
