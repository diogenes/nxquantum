# NxQuantum v0.7 Feature Specification

## Release Theme

Standalone production hardening plus external-operations interoperability contracts.

## Why v0.7 Matters

1. Keep NxQuantum fully useful as a standalone production library.
2. Provide stronger machine-consumable contracts for external telemetry/capability/configuration clients.
3. Protect architecture boundaries so NxQuantum does not absorb orchestration/control-plane product scope.

## Priority Selection (Pareto)

1. **P0: Contract stability for external operations clients**.
2. **P1: Observability exporter hardening and schema governance**.
3. **P2: Integration profiles and standalone production hardening guides**.

Deferred post-v0.7:

1. Central scheduler/queue/retry orchestration engines.
2. Dashboard/UI/RBAC/tenant management capabilities.
3. Fleet-wide workload management semantics.

## Feature Set

### F1. External Operations Integration Contract v1 [P0]

Goal:

1. Expose stable, machine-readable lifecycle and capability contracts for external consumers.

Scope:

1. Add explicit correlation and idempotency envelope fields for lifecycle operations.
2. Ensure typed capability and error contracts remain versioned and externally consumable.
3. Provide deterministic serialization helpers for job/result/error envelopes.

Acceptance:

1. Contract fields are versioned and schema-tested.
2. Equivalent inputs produce equivalent serialized contract output.
3. Public contract changes require explicit version bump/migration note.

### F2. Observability Export and Schema Hardening [P1]

Goal:

1. Keep observability contracts production-grade across profiles and adapters.

Scope:

1. Harden adapter strategy for low-overhead default and exporter-backed integrations.
2. Add schema-governance checks for spans/metrics/log fields and redaction invariants.
3. Preserve deterministic profile behavior (`high_level`, `granular`, `forensics`) with bounded cardinality rules.

Acceptance:

1. Schema tests fail on unversioned breaking telemetry changes.
2. Redaction and cardinality safeguards remain enforced in all profiles.
3. Exporter-backed and no-op/test adapters preserve functional contract behavior.

### F3. Standalone Production Hardening + Integration Profiles [P2]

Goal:

1. Improve standalone production UX while keeping integration with external systems straightforward.

Scope:

1. Publish “standalone production profile” guidance and “external-operations client profile” guidance.
2. Provide canonical examples for:
   - standalone lifecycle execution with observability enabled,
   - machine-consumable capability discovery and typed error handling.
3. Add release evidence checklist items for external integration contract stability.

Acceptance:

1. Standalone and integration profile docs are executable and consistent with public APIs.
2. Reference examples map to existing acceptance scenarios.
3. Release notes explicitly report contract stability status.

## Boundary Contract (Mandatory)

NxQuantum owns:

1. execution, lifecycle, capability, and observability contracts,
2. deterministic typed diagnostics and redaction policy,
3. standalone production usability.

NxQuantum does not own:

1. centralized orchestration policy (global retries/scheduling/queueing),
2. control-plane UX features (dashboards, tenancy, RBAC, incident workflows),
3. external workload fleet management concerns.

## Feature Coverage Plan (v0.7)

1. Provider lifecycle and capability features remain core acceptance paths.
2. Observability contract features remain mandatory for profile and schema checks.
3. Additional contract serialization/integration profile tests map under provider and observability feature suites.

## Delivery Plan

### Milestone O (P0): Contract Envelope Stabilization

1. External operations integration contract versioning and serialization helpers.
2. Correlation/idempotency field stabilization and deterministic tests.

### Milestone P (P1): Observability Schema Governance

1. Adapter/export hardening and schema test expansion.
2. Redaction/cardinality invariant hardening across profiles.

### Milestone Q (P2): Standalone and Integration Profile Enablement

1. Production profile docs and executable examples.
2. Release evidence updates for integration contract stability.

## Success Metrics

1. Zero undocumented breaking changes in lifecycle/capability/observability contracts.
2. Standalone production workflows remain green in default release lanes.
3. External clients can consume deterministic machine-readable contracts without custom parsing hacks.

## Risks and Mitigations

1. Risk: boundary drift into orchestration product behavior.
   Mitigation: enforce boundary ADR and explicit non-goals in feature planning.
2. Risk: telemetry schema churn creates client instability.
   Mitigation: versioned schema checks and release-gate enforcement.
3. Risk: standalone UX regresses while optimizing integration extensibility.
   Mitigation: maintain standalone examples and smoke tests as release blockers.
