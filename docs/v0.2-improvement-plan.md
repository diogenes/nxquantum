# NxQuantum v0.2 Improvement Plan

## Purpose

Provide an execution-ready plan derived from role-based reviews of features, scenarios, APIs, tests, and specs.

## Review Inputs

1. `AGENTS.md`
2. `features/*.feature`
3. `test/features/` and `test/features/steps/*.ex`
4. `docs/*.md` including `v0.2-feature-spec.md`
5. `lib/nx_quantum/*.ex`

## Cross-Agent Review Findings

### Product and Spec Mode

1. Feature scope is strong but needed stricter deterministic acceptance contracts in backend and hybrid training.
2. v0.2 value proposition is high for ML community if differentiation/noise/optimization/kernel tracks stay explicit.

### Architecture Mode

1. Runtime profile and fallback policy needed a concrete contract in code, not only docs.
2. Spec examples required API alignment with existing modules.

### Verification Mode

1. Acceptance coverage map exists, but many tests are placeholders.
2. Deterministic references are present in features and should now be converted into executable assertions incrementally.

### API and DX Mode

1. API facades were present but needed clearer runtime/profile error handling semantics.
2. Public docs needed explicit linkage to the operational plan.

### Docs and Enablement Mode

1. Docs were mostly coherent but lacked a single improvement execution plan document.
2. Agent roles needed stronger sequencing and handoff rules.

## Prioritized Improvement Backlog

## P0 (Immediate)

1. Harden `AGENTS.md` into role-based delivery system with sequence, handoffs, and review gates.
2. Keep backend and hybrid features deterministic, explicit, and testable.
3. Align runtime profile API surface to support typed resolution behavior.
4. Publish a single v0.2 improvement plan and link it from docs index.

## Quality Gate Matrix

| Change type | Required checks | Blocking condition |
| --- | --- | --- |
| Spec/docs only | docs build + link consistency | conflicting contract text |
| API contract | unit + acceptance mapping + typespec/docs update | undocumented contract change |
| Runtime/fallback | contract tests + deterministic error assertions | fallback ambiguity |
| Kernel/perf | property tests + benchmark delta report | regression without benchmark evidence |
| Release batch | `mix quality`, `mix dialyzer`, docs build | any gate failure |

## P1 (Next)

1. Convert top acceptance placeholders into executable tests:
   1. backend profile resolution/fallback.
   2. hybrid deterministic forward/gradient/seed checks.
2. Add unit tests for `NxQuantum.Runtime`.
3. Add deterministic test fixtures for shot/noise behavior.

## P2 (Follow-up)

1. Add benchmark harness for optimizer passes and batch execution.
2. Expand tutorial docs for Axon integration and kernel workflows.

## Development Sequence for Execution

1. Product and Spec mode validates deterministic feature criteria and priority.
2. Architecture mode confirms runtime contracts and module boundaries.
3. API and DX mode implements/aligns public-facing scaffolds.
4. Verification mode creates executable tests for P0 behavior.
5. Docs mode syncs roadmap/spec/testing docs and plan tracking.
6. Release mode validates quality gates and release-readiness checklist.

## Plan Fine-Tuning Pass

Fine-tune decisions applied:

1. Keep explicit runtime profiles and deterministic fallback policies.
2. Keep `mix` as canonical task runner and `mise` for toolchain reproducibility only.
3. Preserve feature ambition but enforce deterministic checks before adding complexity.
4. Split surfaced APIs into stable vs experimental in the v0.2 spec.
