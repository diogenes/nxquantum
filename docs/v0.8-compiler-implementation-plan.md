# v0.8 Compiler Implementation Plan (Production Execution, Qiskit-Pareto)

## Objective

Deliver the minimum high-impact compiler/transpiler feature set required for NxQuantum to compete credibly with Qiskit on production execution workflows, without chasing full ecosystem parity.

This plan operationalizes Phase 15 in `docs/roadmap.md` and is designed for multi-agent handoff under `AGENTS.md`.

## Why This Milestone Exists

Current NxQuantum compiler/transpiler value is deterministic but narrow:

1. Transpiler currently exposes only `:strict` and `:insert_swaps` modes with shortest-path routing.
2. Compiler optimization is pass-based but not yet expressed as production-oriented optimization levels with target-aware policy.
3. Scheduling and calibration-aware compile decisions are not first-class public contracts.

External baseline references informing this milestone:

1. Qiskit transpiler breadth and stage/plugin model (`~/Projects/diogenes/qiskit/qiskit/compiler/transpiler.py`).
2. Qiskit backend target hooks (`~/Projects/diogenes/qiskit/qiskit/providers/backend.py`).
3. Cirq gateset optimization and heuristic routing (`~/Projects/diogenes/Cirq/cirq-core/cirq/transformers/optimize_for_target_gateset.py`, `~/Projects/diogenes/Cirq/cirq-core/cirq/transformers/routing/route_circuit_cqc.py`).

## Pareto Scope (Mandatory)

The following capabilities are mandatory for this milestone:

1. `Target` contract v1:
   - gateset, directed coupling map, durations, error rates, readout metadata, calibration snapshot id.
2. `PassManager` with deterministic `optimization_level` contract:
   - `0` (validity and translation only),
   - `1` (light layout/routing/1q cleanup),
   - `2` (aggressive routing/optimization/scheduling).
3. Routing upgrade:
   - SABRE-like heuristic with lookahead and deterministic seed behavior.
4. Translation and optimization core:
   - target gateset decomposition plus 1q fusion/cancellation and basic commutation-safe reductions.
5. Scheduling:
   - ASAP and ALAP with critical-path and idle-window diagnostics.
6. Calibration-aware policy:
   - cost model can weight depth/duration/error with deterministic profile selection.
7. Explainability:
   - machine-readable compile report with strategy rationale and typed diagnostics.
8. Deterministic cache:
   - compile artifact cache keyed by circuit fingerprint, target fingerprint, and optimization profile.
9. Verification and evidence:
   - feature scenarios, property checks, benchmark deltas, and CI gates.

## Out of Scope (v0.8 P1)

1. Full plugin marketplace compatibility matching Qiskit plugin ecosystems.
2. Pulse-level or provider-native analog program synthesis.
3. Automatic provider-specific policy learning loops.
4. Full cross-vendor transpiler pass parity.

## Mandatory Architecture Constraints

1. Dependency direction remains `Domain <- Application <- Adapters`.
2. Public facade compatibility is preserved unless API and DX Agent explicitly versions a change.
3. Provider-specific compilation metadata stays behind anti-corruption adapters.
4. Deterministic behavior is required for fixed seed + profile + target inputs.
5. Typed failure envelopes are required for unsupported/invalid compile states.

```elixir

  circuit: NxQuantum.Circuit.t(),
  target: NxQuantum.Compiler.Target.t(),
  optimization_level: 0 | 1 | 2,
  routing_strategy: :shortest_path | :sabre_like,
  scheduling_strategy: :none | :asap | :alap,
  calibration_profile: :none | :balanced | :fidelity_first | :latency_first,
  seed: non_neg_integer() | nil

```

```elixir
%{
  circuit: NxQuantum.Circuit.t(),
  report: %{
    optimizer: map(),
    routing: map(),
    scheduling: map(),
    cost_model: map(),
    diagnostics: [map()],
    cache: map()
  }
}
```

## Typed Error Envelope

```elixir
%{
  code:
    :compiler_invalid_target
    | :compiler_unsupported_gate
    | :compiler_topology_violation
    | :compiler_unsatisfied_directionality
    | :compiler_scheduling_failure
    | :compiler_internal_failure,
  stage: :validation | :layout | :routing | :translation | :optimization | :scheduling,
  reason: term(),
  metadata: map()
}
```

## Milestone Structure

### Milestone S1 (P0): Target + PassManager Contract Completion

Deliver:

1. Target model v1.
2. PassManager level contract (`0/1/2`).
3. Stable compile request/result/error envelopes.
4. Failing-first acceptance scenarios for contract behavior.

Review gate:

1. Compile contract is documented and schema-tested.
2. Deterministic behavior is validated for fixed seed/profile.
3. Public API changes are documented in README and docs.

### Milestone S2 (P1): Routing + Translation Competitive Core

Deliver:

1. SABRE-like lookahead routing path with deterministic tie-break.
2. Target-aware decomposition and optimization pipeline.
3. Calibration-aware cost model profile support.

Review gate:

1. SWAP/depth deltas improve against current `:insert_swaps` baseline in benchmark matrix.
2. Semantic equivalence holds under acceptance/property suites.
3. Compile diagnostics explain selected vs rejected strategy paths.

### Milestone S3 (P2): Scheduling + Caching + Release Evidence

Deliver:

1. ASAP/ALAP scheduling support and timing diagnostics.
2. Compile cache for repeated workflow patterns.
3. CI and release gates for compiler behavior/perf evidence.

Review gate:

1. Scheduling metrics appear in compile report.
2. Cache determinism and invalidation behavior are tested.
3. Performance and correctness evidence is release-ready.

## Quality Gates and Metrics

Mandatory quantitative targets:

1. Determinism:
   - identical compile output and report for same input/seed/profile across repeated runs.
2. Routing quality:
   - median SWAP reduction >= 20% vs current shortest-path baseline on scoped benchmark set.
3. Circuit quality:
   - median depth reduction >= 15% at `optimization_level=2` vs `optimization_level=0`.
4. Compile latency:
   - p95 compile time <= 2.0x current baseline for scoped medium circuits (documented benchmark set).
5. Semantic correctness:
   - expectation deltas within configured tolerance envelopes on acceptance/property scenarios.
6. Diagnostics:
   - report schema fields present and versioned; typed errors for all unsupported states.

## Risk Register

1. Scope creep into full parity program.
   - Mitigation: enforce Pareto mandatory set and explicit non-goals.
2. Compile quality improvements regress determinism.
   - Mitigation: deterministic seed/tie-break contracts and property tests.
3. Calibration-aware policy introduces unstable behavior.
   - Mitigation: profile-bound deterministic weighting and explicit report evidence.
4. Multi-agent merge conflicts.
   - Mitigation: disjoint write scopes and strict handoff contracts.
5. Performance wins on one topology degrade others.
   - Mitigation: benchmark matrix includes line, heavy-hex-like, and directed-map families.
